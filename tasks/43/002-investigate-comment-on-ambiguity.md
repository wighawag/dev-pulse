## Description

Update the `investigate()` method in `src/orchestrator.ts` to handle the ambiguity path: when the agent signals ambiguity (via `.whitesmith-ambiguity.md`), comment on the issue with clarification questions instead of creating a PR with task files.

### Current flow
```
investigate() → agent generates tasks → push branch → create PR → label tasks-proposed
```

### New flow
```
investigate() → agent signals ambiguity?
  YES → cleanup local branch → comment on issue with questions → label needs-clarification → return (no PR)
  NO  → existing flow (push branch → create PR → label tasks-proposed)
```

### Changes needed

1. **New label** — Add `NEEDS_CLARIFICATION: 'whitesmith:needs-clarification'` to `LABELS` in `src/types.ts`.

2. **Update `investigate()` in `src/orchestrator.ts`** — After the agent runs:
   - Call `checkForAmbiguity(this.config.workDir)` (from task 43-001).
   - If ambiguous (returns non-null string):
     - **Checkout `main` and delete the local `investigate/<N>` branch.** The branch was already created and checked out before the agent ran — it must be cleaned up on the ambiguity path.
     - Remove the `investigating` label.
     - Add the `needs-clarification` label.
     - Post a comment on the issue using `buildClarificationComment()` (see below).
     - Return early — do NOT push, do NOT create a PR.
   - If not ambiguous (returns null):
     - Continue with the existing flow (verify tasks, push, create PR, etc.)

3. **Comment template** — Add a function `buildClarificationComment(clarificationText: string): string` in `src/prompts.ts` (consistent with other prompt/template functions like `buildInvestigatePrompt`, `buildReviewTaskProposalPrompt`, etc.). The comment should follow this format:
   ```markdown
   🤔 I've analyzed this issue and need clarification before generating implementation tasks:

   <agent's clarification questions from .whitesmith-ambiguity.md>

   ---

   **How to respond:**
   **Edit this issue** — update the description with the missing details.

   I'll automatically re-analyze when the issue description is updated.
   ```

4. **Update `decideAction()` filtering** — Issues with the `needs-clarification` label are automatically excluded from the new-issue scan in `decideAction()` because `NEEDS_CLARIFICATION` is added to `LABELS`, and `decideAction()` uses `noLabels: allDevPulseLabels` where `allDevPulseLabels = Object.values(LABELS)`. This means no manual filtering code is needed in `decideAction()` — just adding the label constant to `LABELS` is sufficient.

5. **Update `decideActionForIssue()`** — When an issue has the `needs-clarification` label, return `{type: 'idle'}`. This is a temporary behavior that will be changed in task 43-003 to return `{type: 'investigate'}` for re-investigation on issue edit. **Note:** Tests for this idle behavior should be aware they will be updated in 43-003.

6. **Update `ensureLabels()`** call — The new label is automatically included because `ensureLabels(Object.values(LABELS))` is called at startup.

### Files to modify

- `src/types.ts` — Add `NEEDS_CLARIFICATION` to `LABELS`.
- `src/prompts.ts` — Add `buildClarificationComment(clarificationText: string): string`.
- `src/orchestrator.ts` — Modify `investigate()` to call `checkForAmbiguity()` and branch on result. Modify `decideActionForIssue()` to handle `needs-clarification` as idle (will change in 43-003).

## Acceptance Criteria

- When the agent signals ambiguity, a comment is posted on the issue with the clarification questions
- The comment is generated by `buildClarificationComment()` in `src/prompts.ts`
- The comment includes clear instructions to edit the issue to provide clarification
- No PR is created when investigation is ambiguous
- No branch is pushed when investigation is ambiguous
- The local `investigate/<N>` branch is deleted and `main` is checked out on the ambiguity path
- The `needs-clarification` label is applied to the issue
- The `investigating` label is removed
- Issues with `needs-clarification` label are skipped during `decideAction()` (automatic via `LABELS` inclusion)
- `decideActionForIssue()` returns `idle` for `needs-clarification` issues (temporary — changed in 43-003)
- The `needs-clarification` label is included in `ensureLabels()` (automatic via `LABELS`)
- Existing investigate flow (non-ambiguous) is unchanged
- Unit tests cover:
  - Ambiguous investigation path: comment posted, no PR, correct labels, branch cleanup
  - Non-ambiguous path still works unchanged
  - `decideActionForIssue` returns idle for needs-clarification issues
  - `buildClarificationComment` output format
- The `createConfig` test helper in `test/orchestrator.test.ts` should include `provider` and `model` fields if any new code paths require them

## Implementation Notes

- The clarification text from `.whitesmith-ambiguity.md` should be trimmed before including in the comment.
- When ambiguous, the branch cleanup sequence is: `await this.git.checkoutMain()` then `await this.git.deleteLocalBranch(branch)`.
- The early return path in `investigate()` should happen inside the `try` block, after the agent runs but before the "Verify task files" step.
- Make sure the agent's clarification text is properly sanitized/trimmed before including in the comment.
